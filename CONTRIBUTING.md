# Contributing

This repository is curated, not exhaustive. New additions should improve the signal-to-noise ratio of the list.

## What Belongs Here

- Canonical books, papers, surveys, courses, and reference material.
- Resources with stable links and a clear learning use case.
- Material that is likely to remain useful beyond the current release cycle or hype wave.

## What Usually Does Not Belong Here

- Product launch posts, benchmark screenshots, or marketing copy.
- Conference schedules, version tables, community size counts, or other fast-aging details.
- Generic homepages when a paper, official course page, or direct documentation link exists.
- Duplicate resources that teach roughly the same thing with less clarity.

## Required Metadata

Each new README entry should include:

- A cost tag: `[Free]` or `[Paid]`
- A level tag: `[Beginner]`, `[Intermediate]`, or `[Advanced]`
- A type tag: `[Book]`, `[Course]`, `[Paper]`, `[Survey]`, or `[Reference]`
- One short sentence explaining why the resource is worth including

## Where to Add Things

- Add only broadly useful, first-choice resources to the main `README.md`.
- Add narrower or second-tier material to the relevant file under `docs/`.
- If no topic guide exists yet, add one under `docs/README.md` instead of overloading the main README.
- When moving material out of the main README, keep the short path there and put the extra depth in `docs/`.

## Curation Rules

- Prefer primary sources.
- Prefer free or open-access material when possible.
- Keep the README selective. Aim for about 3 to 6 items per section.
- Avoid unsupported claims such as "best", "state of the art", or ranking language.
- If a section grows beyond the README, move the long tail into a separate document and link to it.

## Workflow

1. Create a focused branch from the default branch.
2. Keep changes small and explain the curation rationale in the pull request.
3. Verify links before opening the pull request.
4. Update documentation when the selection logic changes.

## Pull Request Checklist

- Links were checked
- Resource metadata is complete
- Descriptions are concise and specific
- README scope stayed focused
- No secrets or credentials were committed

## Commit Style

Use clear commit messages describing intent and scope.
